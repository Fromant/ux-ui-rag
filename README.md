# RAG-система для учебника по дискретной математике

Локальная система поиска и тренажёр для учебника по дискретной математике (DM2024.pdf). Работает полностью офлайн, без использования LLM.

**Важно:** Учебник и сгенерированные PNG-файлы **не** хранятся в репозитории. При первом запуске нужно положить PDF в `books/DM2024.pdf` и запустить `setup.sh`.

## Возможности

- **Поиск по учебнику** - BM25 полнотекстовый поиск по секциям учебника
- **Тренажёр/Quiz** - режим проверки знаний с отложенным показом контекста
- **Просмотр страниц** - PNG-изображения страниц учебника с навигацией
- **Навигация** - загрузка дополнительных страниц в обе стороны

## Требования

- Docker
- CPU-only
- ~2.5GB RAM (model requires ~500MB)
- ~500MB disk space (includes model cache)

## Быстрый старт

```bash
# 1. Положить учебник (PDF) в books/DM2024.pdf
#    Название должно быть точно DM2024.pdf

# 3. Собрать и запустить Docker
```

## Архитектура

### Backend (FastAPI)

**main.py** - основное приложение:
- `POST /api/search` - поиск по учебнику
- `GET /api/pages/{page_num}` - получить диапазон страниц
- `POST /api/simulator/next` - следующий вопрос
- `POST /api/simulator/validate` - проверить ответ
- `GET /data/pages/{filename}` - отдать PNG страницу

**app/search/simple_search.py** - BM25 поиск:
- Использует библиотеку `rank-bm25`
- Индекс строится из заголовков секций и ключевых слов
- Возвращает top-k результатов с оценками релевантности

**app/search/pdf_indexer.py** - индексация:
- Извлекает секции вида `1.2.3. Название секции`
- Очищает заголовки от артефактов (2/2), обрывков фраз
- Вытаскивает ключевые слова из текста страницы
- Сохраняет в JSON для быстрой загрузки

**app/models/answer_validator.py** - проверка ответов:
- Использует sentence transformers для семантического анализа
- Модель `paraphrase-multilingual-MiniLM-L12-v2` (поддерживает русский)
- Сравнивает эмбеддинги ответа студента с текстом секции
- Возвращает оценку схожести и уровень уверенности

**app/processors/pdf_to_images.py** - конвертация:
- Использует PyMuPDF (fitz) для рендеринга PDF
- Сохраняет страницы как PNG с DPI=150
- Именование: `page_0001.png`, `page_0002.png`, ...

### Frontend

**templates/index.html** - страница поиска:
- Вертикальный скролл результатов
- Кнопки "Предыдущие 5" / "Следующие 5" для загрузки страниц учебника
- Показ диапазона загруженных страниц
- Click-to-expand для просмотра страниц

**templates/simulator.html** - тренажёр:
- Вопрос показывается БЕЗ контекста
- После ответа студента показываются страницы учебника
- Оценка ответа через семантическую схожесть (sentence transformers)
- Разные уровни уверенности: high/medium/low/very_low
- Развёрнутая обратная связь в зависимости от уверенности

## Как это работает

### Поиск

1. Пользователь вводит запрос (например, "графы")
2. BM25 вычисляет релевантность для каждой секции
3. Возвращаются top-k результатов (по умолчанию 3)
4. Для каждого результата показывается главная страница +-2 (всего 5)
5. Frontend позволяет листать вверх/вниз кнопками +-5

### Индексация

1. PDF открывается через PyMuPDF
2. Регуляркой ищутся заголовки вида `1.2.3. Название`
3. Заголовки очищаются от артефактов (2/2), обрывков, случайных имён
4. Из текста страницы извлекаются ключевые слова (частотный анализ)
5. Результат сохраняется в JSON (~400 чистых секций)

### Конвертация PDF в изображения

1. Каждая страница PDF рендерится в PNG через PyMuPDF
2. DPI=150 обеспечивает баланс качества и размера
3. 1738 страниц → ~167MB PNG-файлов

## Дизайн решения

### Почему PNG, а не текст?

Исходный PDF имеет проблемы с извлечением текста (нет пробелов между словами, сложный формат текста, использование греческого алфавита, множество таблиц). Использование готовых PNG-изображений:
- Сохраняет оригинальное форматирование
- Не требует OCR (который плохо работает с русским+английским техническим текстом)
- Проще в отображении

### Почему BM25, а не LLM?

Требования включали работу без LLM для поиска:
- Не нужно GPU для поиска
- Быстрее и проще развёртывание
- Работает офлайн
- Меньше ресурсов

**Для проверки ответов** используется sentence-transformers (не LLM):
- Маленькая модель (~120MB)
- Быстрая работа (~100-200ms на ответ)
- Понимает семантику, а не только ключевые слова
- Работает офлайн на CPU

### Почему контекст ПОСЛЕ ответа?

В тренажёре важно, чтобы студент сначала попробовал ответить сам, а потом посмотрел материал. Это активное обучение, а не пассивное чтение.

## Разработка

```bash
# Локальный запуск (без Docker)
pip install -r requirements.txt
python main.py

# Пересоздание индекса
python -m app.search.pdf_indexer

# Переконвертация PDF
python -m app.processors.pdf_to_images
```

## Размеры

- Исходный PDF: ~30MB
- PNG изображения: ~167MB (1738 страниц)
- Индекс секций: ~150KB (~400 чистых записей)
- Модель sentence-transformers: ~120MB (кэшируется)
- Docker образ: ~1.2GB (включает torch и модель)

## Ограничения

- Семантическая валидация работает хорошо, но не идеальна (порог 0.35 cosine similarity)
- Модель требует ~500MB RAM и добавляет ~100-200ms на проверку ответа
- Индексация отфильтровывает слишком много секций (~400 из ~2000+)
- Все секции индексируются, даже оглавление/приложения
